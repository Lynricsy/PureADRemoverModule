## 2. Root Module Shell Actions

`module/action.sh`:

- Actions default to `status`; real mutation is only exposed through explicit `apply-profile-execute <profile>` and `profile-restore-execute`.
- Profile arguments and env-derived paths are passed as quoted argv to `puread-cli`, not interpolated into shell command strings:
  - `"$PUREAD_CLI" apply-profile "$PUREAD_PROFILE_NAME" "--$PUREAD_APPLY_MODE" --rules "$PUREAD_RULES_DIR" --root "$PUREAD_PROFILE_ROOT" --module-root "$PUREAD_MODDIR"`
  - `"$PUREAD_CLI" profile-restore "--$PUREAD_RESTORE_MODE" --module-root "$PUREAD_MODDIR"`
- Extra action arguments are rejected with `puread_require_no_extra_arg`.
- The shell-side profile ledger bootstrap rejects symlink leaf/parent, non-directory parent, and non-file ledger before creating `state/profile-actions.jsonl`, then applies `0700` directory and `0600` file permissions.
- Residual note: shell-side ledger creation is not atomic `O_NOFOLLOW`; however Rust profile ledger read/rewrite paths also enforce module-root containment, parent symlink rejection, leaf symlink rejection, and `O_NOFOLLOW` before reading or writing. This is recorded as a non-blocking defense-in-depth note, not a blocker.

`module/service.sh`:

- Host execution is dry-run only outside Android.
- Android daemon default is `PUREAD_DRY_RUN=1`; non-dry-run requires explicit `PUREAD_DRY_RUN=0`.
- No DNS/hosts/proxy/iptables/AdGuardHome/ad_reward/root hiding code was found.

Rust command execution boundaries:

- `crates/puread-android/src/command_runner/validation.rs` rejects empty values, whitespace, NUL, option-like tokens, `..`, and shell metacharacters before constructing Android command argv.
- `pm`, `cmd appops`, `settings`, `getprop`, `chcon`, and related adapters construct typed `CommandInvocation` argv arrays, not shell strings.
- `settings` rejects `private_dns_mode`, `private_dns_specifier`, any key containing `dns`, `host`, or `proxy`, and only allows the ROM-ad settings allowlist.
- Profile apply execute acquires `GlobalLock`, preflights profile ledger safety for non-file actions, then executes typed file/profile actions.
- Profile restore execute acquires `GlobalLock`, reads ledger through no-follow/symlink-checked paths, restores final-state entries, and rewrites only after successful restore actions.
