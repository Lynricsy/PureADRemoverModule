## Root Cause
- `ComponentActionKey` only used `rule_id`, `package`, and `component`.
- Two component records that differed only by `user_id` folded into one final-state restore action.
- `rewrite_restored` then received the folded action indexes and could mark an un-restored user record as restored.

## Red Test
- Command:
  `cargo test -p puread-cli --test cli_profile_restore cli_profile_restore_keeps_component_final_state_per_user_id -- --nocapture`
- Before fix result:
  - test failed.
  - report assertion observed `restored = Number(1)` but expected `2`.
  - earlier runner-log assertion showed only one `pm enable --user 10 ...` call, proving user 0 was not actually restored.

## Fix
- Added `user_id` to `ComponentActionKey`.
- Parsed `user_id` from component JSONL records as `u32`.
- The final-state fold now treats user 0 and user 10 as distinct logical restore keys.

## Regression Test Coverage
- `cli_profile_restore_keeps_component_final_state_per_user_id` writes two component records whose only logical difference is `user_id`.
- It executes real CLI `profile-restore --execute` via the integration-test binary.
- It asserts:
  - JSON report has `restored = 2`.
  - runner log contains exactly one `pm enable --user 0 ...`.
  - runner log contains exactly one `pm enable --user 10 ...`.
  - rewritten ledger contains exactly two restored records.
  - each user-specific ledger line has its own `restore_status=restored`.
