## Blocking security issue

### 1. Profile state ledger and lock paths follow symlinks before/while opening state files

Severity: HIGH for module-state integrity and recoverability.

Affected code:

- `crates/puread-cli/src/profile_execute/profile_runtime.rs:57-65`
  - `JsonlProfileLedger::append()` uses `std::fs::create_dir_all(parent)` and `OpenOptions::new().create(true).append(true).open(&self.path)`.
  - This follows symlinked parent directories and a symlinked final `profile-actions.jsonl`.
- `crates/puread-cli/src/lock.rs:22-31`
  - `GlobalLock::acquire()` uses `fs::create_dir_all(parent)` and plain `OpenOptions::open(path)`.
  - This follows a symlinked `run` parent or final `puread.lock`.
- `crates/puread-cli/src/profile.rs:76-84`
  - `apply-profile --execute` derives the lock and profile ledger from user-supplied `--module-root`, then executes Android profile mutations after ledger append.

Risk chain:

1. An attacker or buggy prior run controls or tampers with the module state path, for example:
   - `/data/adb/modules/puread/state` is a symlink to another writable directory, or
   - `/data/adb/modules/puread/state/profile-actions.jsonl` is a symlink to another file, or
   - `/data/adb/modules/puread/run` / `puread.lock` is a symlink.
2. User runs an explicit Android profile mutation:
   - `puread-cli apply-profile component --execute --rules rules/components --root / --module-root /data/adb/modules/puread`
3. The lock open and profile ledger append follow the symlink.
4. `AndroidProfileExecutor::operation()` considers the append successful because `ProfileLedgerSink::append()` returned `Ok(())`.
5. Mutation proceeds:
   - component path: `pm hide` capability attempt and `pm disable-user`
   - appops path: `cmd appops set`
   - ROM settings path: `settings put`
6. The restore record may be written outside the intended module state ledger, may be attacker-controlled, or may be unavailable to the normal restore/report path. This breaks the key safety invariant that mutation is recoverable from module-local state.

Concrete reproduction sketch:

```sh
ROOT="$(mktemp -d)"
MODULE="$ROOT/module"
OUTSIDE="$ROOT/outside"
mkdir -p "$MODULE" "$OUTSIDE"
ln -s "$OUTSIDE" "$MODULE/state"
PUREAD_TEST_PROFILE_RUNNER=1 \
PUREAD_TEST_PROFILE_RUNNER_LOG="$ROOT/runner.log" \
cargo run -q -p puread-cli -- \
  apply-profile component --execute \
  --rules rules/components \
  --root tests/fixtures/android-fs \
  --module-root "$MODULE"

# Expected secure behavior: fail before mutation because state parent is a symlink.
# Current risk: profile ledger append follows "$MODULE/state" to "$OUTSIDE/profile-actions.jsonl",
# then pm hide / pm disable-user can be issued through the runner.
test -f "$OUTSIDE/profile-actions.jsonl"
grep -E "pm hide|pm disable-user" "$ROOT/runner.log"
```

A similar lock-path reproduction uses `ln -s "$OUTSIDE" "$MODULE/run"` before `apply-profile --execute`; the lock file is opened through the symlink instead of being rejected as an unsafe module-state path.

Why this is blocking:

- The project safety contract requires real mutation to be conservative and recoverable.
- The reviewed profile execution path correctly requires ledger append before mutation, but the ledger sink itself is not protected against state path symlink escape.
- The user explicitly requested review of lock/state file permissions, paths, and concurrency risk.
- Existing tests cover injected ledger failure (`PUREAD_TEST_PROFILE_LEDGER_FAIL`) but do not cover symlinked module state paths or final ledger symlink.

Minimum fix:

- Replace profile ledger append with secure module-state file I/O:
  - Validate `module_root` is absolute for real execution.
  - Ensure `state/` parent chain under `module_root` is no-follow and not symlinked.
  - Open `state/profile-actions.jsonl` with final-component no-follow semantics.
  - Prefer `O_NOFOLLOW | O_CLOEXEC | O_APPEND | O_CREAT`; use mode `0600` or `0640`.
  - Reject symlinked final ledger path instead of following it.
- Replace lock acquisition path setup/open with secure no-follow I/O:
  - Ensure `run/` parent chain under `module_root` is no-follow.
  - Open final `puread.lock` with no-follow and a restrictive mode.
  - Reject symlinked override lock paths, or only allow overrides after explicit no-follow validation.
- Add targeted tests:
  - `apply-profile --execute` rejects `module_root/state` parent symlink and does not issue `pm hide`, `pm disable-user`, `cmd appops set`, `settings put`, or XML write.
  - `apply-profile --execute` rejects final `state/profile-actions.jsonl` symlink before mutation.
  - `apply-profile --execute` rejects `module_root/run` and final `run/puread.lock` symlinks before mutation.
  - Assertions should verify runner log contains only read-only probes, or no commands if lock validation fails first.

## Passed checks

### Restore --execute target and backup safety

Result: PASS.

Evidence:

- `crates/puread-cli/src/restore.rs:50-63` acquires the global lock before reading/restoring records and only applies clear attempts after restore attempts are collected.
- `crates/puread-cli/src/restore.rs:71-87` records failed restore attempts as failed in the report and attempt list.
- `crates/puread-cli/src/restore_fs.rs:27-35` enforces root-bound and parent-chain checks through `RestoreRoot`.
- `crates/puread-cli/src/restore_fs.rs:96-108` calls `root.guard_existing_path(path)` before `remove_dir_all` or `remove_file`.
- `crates/puread-cli/src/restore_fs.rs:116-118` uses the same guarded `remove_placeholder()` for directory restore replacement.
- `crates/puread-cli/src/restore_fs.rs:147-158` ensures parent path then rejects final symlink and opens with `O_NOFOLLOW`.
- `crates/puread-cli/src/restore_fs.rs:255-267` rejects final symlink.
- `crates/puread-core/src/restore_ledger/store.rs:115-136` only removes succeeded attempts, preserves failed attempts, and returns an error if ledger rewrite fails.
- Tests in `crates/puread-cli/tests/cli_execute_security.rs:217-266` and support cases in `tests/support/cli_restore_adversarial.rs:18-199` cover target symlink, target parent symlink, directory restore parent symlink, explicit `RemovePlaceholder` parent symlink, backup symlink, backup parent symlink, backup path escape, missing parent recreation, and mode restore.

Residual note:

- `restore_fs.rs` uses `remove_dir_all()` after no-follow parent/final checks. That still has theoretical TOCTOU exposure between check and deletion if an attacker can swap the final path after validation. Current tests cover parent symlink attacks requested for T20; a stronger future hardening would use descriptor-relative deletion, but I did not classify this as the blocker in this final review because the explicit required parent-symlink and final-symlink checks are present and tested.
