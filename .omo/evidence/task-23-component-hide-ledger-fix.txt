Task: task-23 component pm hide ledger/report fix
Timestamp: 2026-06-18T18:55:43+08:00

Why this was fixed:
- Android component profile execution uses ledger-before-mutation. The pre-mutation persisted
  component record cannot truthfully claim that a later `pm hide` mutation succeeded.
- Before this fix, `TryHide` records were persisted with `hide_status:"applied"` before running
  `pm hide`, while the returned report could later show `hide_status:"skipped_unavailable"`.
  A persisted `applied` record drives restore through `pm unhide`, which is wrong when hide was
  unavailable or failed.

What changed:
- `crates/puread-android/src/profiles/executor/appops_components.rs`
  - Changed the pre-mutation ledger status for `PmHidePolicy::TryHide` from `applied` to
    `skipped_unavailable`.
  - Kept the existing report path: after the `pm hide` attempt, the returned operation record still
    reports `applied` only when the command succeeds and `skipped_unavailable` when it fails.
  - Existing restore logic already gates `pm unhide` on persisted `hide_status == applied`, so a
    conservative persisted record skips `pm unhide` while still continuing component enable.
- `crates/puread-android/tests/appops_components.rs`
  - Added ledger assertions for both unavailable runner errors and non-zero `pm hide` exits.
  - Added restore coverage for a persisted `skipped_unavailable` component record to prove it calls
    `pm enable` without calling `pm unhide`.
  - Kept the existing ledger failure test covering mutation blocking.
