Blocking issue

1. crates/puread-android/src/profiles/executor/appops_components.rs:63-87 writes a component ledger record before attempting pm hide, using hide_status_for_policy(TryHide) which returns skipped_unavailable. If pm hide later succeeds, only the returned report record is rebuilt with hide_status=applied; the already-appended persistent ledger record remains skipped_unavailable.

Why this blocks approval:
- restore_component only runs pm unhide when the persisted record has original_hidden=visible and hide_status=applied at crates/puread-android/src/profiles/executor/appops_components.rs:96-99.
- The real CLI profile-restore path reads persisted JSONL entries from state/profile-actions.jsonl, not the apply-profile in-memory report record.
- Therefore a successful pm hide can leave a package hidden after profile-restore because the durable restore ledger says hide was skipped.
- The existing passing test component_profile_applies_and_restores_available_pm_hide restores from apply.record, not from ledger.records()[0], so it does not prove the real persistent restore path.

Required evidence to clear:
- Add or update a test proving that when pm hide succeeds, the persistent ledger record contains "hide_status":"applied".
- Add or update a CLI/integration-level test where apply-profile --execute persists a successful TryHide component record, then profile-restore --execute reads the JSONL ledger and invokes pm unhide.
- Adjust implementation so durable ledger and report record agree for pm hide success and failure/unavailable paths without allowing mutation before a recoverable ledger entry exists.
