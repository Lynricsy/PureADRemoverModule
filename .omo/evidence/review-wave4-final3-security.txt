Check 1: profile-restore --execute obtains global lock before reading ledger and before rewrite; cannot overwrite entries appended after an old snapshot.
Result: PASS.
Evidence:
- crates/puread-cli/src/profile_execute/profile_restore.rs:40-47 builds the profile ledger path, then for execute obtains lock_path and GlobalLock::acquire before ledger::read_entries.
- crates/puread-cli/src/profile_execute/profile_restore.rs:47-55 reads entries after the lock, computes restore actions from that snapshot, then calls ledger::rewrite_restored while the _lock guard is still in scope.
- crates/puread-cli/src/profile.rs:79-96 shows apply-profile --execute also acquires the same GlobalLock before profile ledger preflight and before execute_android_profile_surface can append profile records.
- crates/puread-cli/tests/cli_profile_restore.rs:212-214 and tests/support/cli_profile_restore.rs:163-170 cover lock contention preventing even ledger read by deleting the ledger while a held lock should fail first.
Assessment:
- Cooperative PureAD profile writers and restore now serialize on the same module global lock, so restore no longer reads an old snapshot before lock acquisition and then truncates over later command-appended records.
