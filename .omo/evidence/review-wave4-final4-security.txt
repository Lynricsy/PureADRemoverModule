2. `profile-restore --execute` still locks before reading, and folding/rewrite does not drop concurrent/after/different-user records.
- PASS, with the expected project-lock precondition.
- Execute path:
  - `profile_restore.rs:45-47` resolves lock path, acquires `GlobalLock`, then reads profile ledger.
  - Dry-run path reads without lock at `profile_restore.rs:41-43`, but does not mutate.
- Lock contention is tested at `crates/puread-cli/tests/cli_profile_restore.rs:269-273` and helper `crates/puread-cli/tests/support/cli_profile_restore.rs:163-170`, where the ledger file is removed while the lock is held; restore fails on lock before reading or runner mutation.
- Rewrite behavior:
  - `ledger.rs:60-92` only rewrites when all restore actions succeeded; failures leave the ledger untouched.
  - It emits every entry from the locked read set in original order and only injects `restore_status` for `restored_indexes`.
  - Final-state folding returns indexes for matching component keys only; different user keys are preserved independently by the key above.
  - This preserves later/different records already present in the ledger read under lock.
- Concurrency precondition:
  - Production CLI apply path acquires the same global lock before profile ledger append (`crates/puread-cli/src/profile.rs:79-83`).
  - Restore uses the same lock before reading/rewrite (`profile_restore.rs:45-55`).
  - Therefore no production CLI writer should append between restore read and rewrite. External writers bypassing the module lock are outside the reviewed production surface.
