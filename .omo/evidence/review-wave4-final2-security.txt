## 阻塞安全问题

### 1. `profile-restore --execute` 在获取全局锁前读取 profile ledger，可能丢失并发 apply 写入的恢复记录

严重性：HIGH，影响模块状态账本完整性与可恢复性。

代码证据：

- `crates/puread-cli/src/profile_execute/profile_restore.rs:39-45`
  - 先执行 `ledger::read_entries(module_root, path.as_path())?`
  - 之后才执行 `GlobalLock::acquire(lock.as_path())?`
- `crates/puread-cli/src/profile_execute/profile_restore.rs:53`
  - restore 成功后用读取到的旧 `entries` 调用 `ledger::rewrite_restored(...)`
- `crates/puread-cli/src/profile_execute/profile_restore/ledger.rs:59-86`
  - `rewrite_restored()` 会用旧 entries 生成完整输出并 `truncate(true)` 重写账本。

风险链：

1. `profile-restore --execute` 启动并在未持锁状态读取旧 ledger，例如只读到记录 A。
2. 另一个 `apply-profile --execute` 先拿到同一 `puread.lock`，追加记录 B，并执行真实 Android mutation。
3. apply 释放锁。
4. restore 才拿到锁，按旧快照恢复 A，并 `truncate` 重写 ledger。
5. 记录 B 被丢弃，但 B 对应的 AppOps、component、ROM settings/XML mutation 已经发生。

这会让后续 `profile-report` / `profile-restore` 找不到 B 的恢复记录，违反项目“真实修改必须写入恢复账本、可恢复”的安全契约。正确顺序应为：在 `execute` 模式下先计算并获取 lock，再在锁内读取 ledger、恢复、重写 ledger。dry-run/report 读取可保持无锁或只读策略。

所需证据：

- 增加并通过针对性并发/交错测试，证明 `profile-restore --execute` 在持锁后读取账本。
- 测试应覆盖 restore 读取前后有 apply 写入的场景，确认新记录不会被旧 entries 的 `truncate` 重写丢失。

### 2. `pm hide` 成功时持久 profile ledger 仍记录 `skipped_unavailable`，成功隐藏不可由持久账本恢复

严重性：HIGH，影响 component profile 的恢复完整性。

代码证据：

- `crates/puread-android/src/profiles/executor/appops_components.rs:63-73`
  - `hide_status_for_policy(PmHidePolicy::TryHide)` 预先生成 `HideApplyStatus::SkippedUnavailable`。
  - `self.applied(&record)?` 在执行 `pm hide` 前把该记录写入持久 ledger。
- `crates/puread-android/src/profiles/executor/appops_components.rs:74-87`
  - 之后才执行 `self.apply_hide_policy(...)`。
  - 如果 `pm hide` 成功，返回给 CLI report 的 transient record 会变成 `hide_status:"applied"`。
  - 但持久 ledger 仍是 mutation 前写入的 `hide_status:"skipped_unavailable"`。
- `crates/puread-android/src/profiles/executor/appops_components.rs:95-100`
  - restore 只有在 `original_hidden == Visible && hide_status == Applied` 时才调用 `pm unhide`。

测试证据缺口：

- `crates/puread-android/tests/appops_components.rs:50-155` 覆盖了 hide 不可用/失败时持久 ledger 与 report 都为 `skipped_unavailable`。
- `crates/puread-android/tests/appops_components.rs:179-221` 覆盖了 transient `apply.record` 成功时能触发 `pm unhide`，但没有断言持久 `ledger.records()` 在成功 `pm hide` 后也保存 `hide_status:"applied"`。

风险链：

1. component 规则执行，持久 ledger 先记录 `hide_status:"skipped_unavailable"`。
2. 随后的 `pm hide` 实际成功，package 被隐藏。
3. 后续 `profile-restore --execute` 从持久 ledger 读取 `skipped_unavailable`。
4. restore 不调用 `pm unhide`，只执行 component enable。
5. package hidden 状态残留，真实 mutation 不可完整恢复。

这同时影响用户要求复查的第 3 项“apply-profile --execute 写入 module-local recoverable record before mutation”和第 5 项“component pm hide persistent ledger”语义：当前记录虽然在 mutation 前写入，但在 `pm hide` 成功路径不是 recoverable record。

所需证据：

- 增加并通过测试：当 `pm hide` 成功时，持久 profile ledger 中的 component 记录必须能驱动 restore 调用 `pm unhide`，或生产逻辑必须避免执行无法预先准确记账的 `pm hide` mutation。
- 测试应直接检查 `ledger.records()`，不能只检查 transient `apply.record`。
