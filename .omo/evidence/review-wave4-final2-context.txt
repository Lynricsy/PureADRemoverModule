## Current blockers

### Blocker 1: successful `pm hide` is not recoverable from durable profile ledger

Evidence:

- `.omo/evidence/review-wave4-final2-quality.txt`: FAIL.
- `.omo/evidence/review-wave4-final2-security.txt`: FAIL.
- Current source:
  - `apply_component()` persists a component record before `pm hide`.
  - For `PmHidePolicy::TryHide`, the persisted pre-mutation record uses `hide_status:"skipped_unavailable"`.
  - If `pm hide` later succeeds, only the returned transient report record is rebuilt with `hide_status:"applied"`.
  - `profile-restore --execute` reads the durable JSONL ledger, not the transient apply report.
  - `restore_component()` only calls `pm unhide` when durable `hide_status == applied`.

Impact:

- A successful `pm hide` can leave the package hidden after profile restore.
- This violates the project invariant that real mutations must be recoverable from module-local ledger state.

Required closure evidence:

- Add/update a test proving successful `pm hide` produces a durable ledger record that drives `pm unhide` during restore, or change production behavior so `pm hide` is not executed unless its recoverable durable record can be written.
- The test must inspect persisted `ledger.records()` or the CLI `state/profile-actions.jsonl`, not only transient `apply.record`.
- Rerun and record:
  - `cargo test -p puread-android --test appops_components -- --nocapture`
  - `cargo test -p puread-cli --test cli_profile_restore -- --nocapture` if CLI restore path is affected.

### Blocker 2: `profile-restore --execute` reads profile ledger before acquiring global lock

Evidence:

- `.omo/evidence/review-wave4-final2-security.txt`: FAIL.
- Current source:
  - `profile_restore.rs` reads entries before `GlobalLock::acquire()`.
  - On success, `rewrite_restored()` truncates and rewrites based on the pre-lock entries snapshot.

Risk:

1. `profile-restore --execute` reads old entries without holding lock.
2. Concurrent `apply-profile --execute` obtains the same lock first, appends a new record, and performs a real mutation.
3. Restore then obtains lock and rewrites from the old snapshot.
4. The new apply record can be lost while the mutation remains live.

Impact:

- Module-local recovery ledger can lose real mutation records.
- This violates the same recoverability invariant.

Required closure evidence:

- Move execute-mode profile ledger read inside the global lock.
- Add an interleaving/concurrency test or equivalent controlled test proving a record appended before restore obtains the lock is not dropped by stale `entries` rewrite.
- Rerun and record `cargo test -p puread-cli --test cli_profile_restore -- --nocapture`.
