# task-20 profile restore lock order fix evidence

## Scope
- Fixed only CLI profile restore lock ordering.
- Touched implementation/test files:
  - `crates/puread-cli/src/profile_execute/profile_restore.rs`
  - `crates/puread-cli/tests/cli_profile_restore.rs`
  - `crates/puread-cli/tests/support/cli_profile_restore.rs`
- Did not modify Android appops/components files.
- Did not commit, push, or set git config.

## Root Cause
`profile-restore --execute` read `state/profile-actions.jsonl` before acquiring `GlobalLock`.
If another `apply-profile --execute` appended to the profile ledger while holding the lock after restore had read stale entries, restore could later truncate/rewrite the ledger from stale entries and drop the new append.

## Fix
`profile-restore --execute` now:
1. computes the profile ledger path;
2. resolves and acquires the global lock;
3. reads the profile ledger;
4. restores pending profile records;
5. rewrites restored records.

Non-execute dry/report behavior remains read-only and does not acquire the global lock.
